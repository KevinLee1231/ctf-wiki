# Weird EVM

## 题目简述

题目运行在修改过的 geth 上：常规 EVM opcode 的字节编号被重新排列，同时加入了 ed25519 与 BN256G1（压缩点表示）各三种预编译操作。使用标准编译器生成的字节码会按错误语义执行，因此第一步是恢复 opcode 映射，或直接使用题目配套的修改版 Solidity 编译器。

新增预编译合约的地址和功能可以从官方 Solidity 封装中直接确认：

| 地址 | 曲线 | 功能 |
| --- | --- | --- |
| `0x15` | ed25519 | 点加 |
| `0x16` | ed25519 | 标量乘 |
| `0x17` | ed25519 | 哈希到点 |
| `0x18` | BN256G1 | 点加 |
| `0x19` | BN256G1 | 标量乘 |
| `0x1a` | BN256G1 | 哈希到点 |

`Game` 合约生成一个 4 位整数 $x$、ed25519 点 $G_2$ 和 BN256G1 点 $H_2$，公开 $xG_2$，要求提交 $xH_2$，并给出“两个不同群中的离散对数是同一个整数”的 DLEQ 证明。`check()` 还要求调用者已有运行时代码，因此最终调用必须来自合约而不是 EOA。

## 解题过程

### 恢复修改后的执行环境

可以对每个单字节 opcode 构造一次 `debug_traceCall`，读取第一条 trace 的 `op` 字段，从而枚举 0 到 255 的实际映射。仓库中的提示脚本正是逐字节执行这一过程。另一种静态路线是在 geth 中搜索 opcode 的 `String` 映射，并在新增预编译实现中搜索 `RequiredGas` 与 `Run`，确认输入输出格式。

恢复映射后，可以修补反编译器，也可以直接使用题目提供的修改版 `solc` 编译官方求解合约。仅替换 RPC 地址并运行普通 Web3 脚本并不够；部署字节码必须符合这台 geth 的 opcode 编码。

### 理解跨群 DLEQ 验证

`newChallenge()` 令 `bits = 4`，生成 $0\leq x<16$，并返回 $x$；同时保存：

$$
xG_2=[x]G_2.
$$

验证者希望收到 $xH_2=[x]H_2$。证明把 $x$ 分解为

$$
x=\sum_{i=0}^{3} b_i2^i,\qquad b_i\in\{0,1\}.
$$

对每一位同时构造两个群上的承诺 $C_i^G$、$C_i^H$。验证器首先检查：

$$
\sum_{i=0}^{3}2^iC_i^G=xG_2,
\qquad
\sum_{i=0}^{3}2^iC_i^H=xH_2,
$$

随后用两分支 Schnorr 型环签名证明每对承诺表示同一个比特，而且该比特只能是 0 或 1。这个构造来源于 Monero Research Lab 的 [MRL-0010：Discrete logarithm equality across groups](https://www.getmonero.org/resources/research-lab/pubs/MRL-0010.pdf)：论文的关键思想就是把共同离散对数按位分解，在两个群中分别承诺每一位，再对每一位给出一致的 0/1 环证明。读者不需要依赖论文才能复现本题，合约中的两个加权和与两分支哈希检查已经完整实现了验证逻辑。

### 构造官方简化证明

本题的 `newChallenge()` 会直接把 $x$ 返回给调用合约，而且官方解法不使用 blinder。对第 $i$ 位：

- 若 $b_i=1$，令 $C_i^G=G_2$、$C_i^H=H_2$；
- 若 $b_i=0$，令两个承诺分别为对应群的无穷远点；
- 将 `a0/a1/b0/b1` 全部设为 1，再按 `Game.check()` 的哈希顺序补出 `eG` 与 `eH`。

这样加权和自然等于 $xG_2$ 和 $xH_2$，而每一位的真实分支可直接满足验证，另一分支则通过 Fiat–Shamir 哈希闭环模拟。

官方 `Solu` 合约在一笔调用中完成以下流程：

```solidity
uint256 x = game.newChallenge();
uint256 G2 = game.G2();
uint256 H2 = game.H2();

// 按 x 的 4 个比特填写 CG、CH、eG、eH 和响应值
game.check(bn.scalarMult(x, H2), proof);
```

部署 `Solu(GameAddress)` 后调用 `solve()`。由于调用来自已部署合约，`extcodesize(caller()) != 0` 成立；证明通过后 `Game.isSolved()` 被置为 `true`，即完成题目验证。

## 方法总结

- 核心技巧：恢复自定义 opcode 映射，调用新增曲线预编译，并构造跨 ed25519/BN256G1 的同离散对数证明。
- 识别信号：标准字节码在目标链上表现异常、trace 中 opcode 名称与字节不一致、低地址出现非标准预编译合约时，应先还原执行环境而不是直接调试业务合约。
- 复用要点：跨群 DLEQ 可以把共同标量按位分解，再用两个群上的加权承诺和逐位 0/1 环证明建立一致性；具体题目若泄露标量或削弱 blinding，可显著简化证明构造。
