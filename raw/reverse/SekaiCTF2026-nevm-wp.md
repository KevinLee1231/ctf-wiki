# nevm

## 题目简述

题目给出四个约 80 MB 的 EVM 部署字节码 `challenge_1.bin` 至 `challenge_4.bin`，以及各自的 16 字节目标密文。每份合约都在一个带自解密 handler 的小型寄存器 VM 中，以不同顺序级联 AES-128、双路 DES 和 SPECK-128/128。需要从字节码本身恢复每层算法及全部参数，逆向得到四个 16 字节明文块，拼接后才是完整 flag。

运行环境不是标准 EVM：补丁为 opcode `0x0c` 增加 `DECRYPT_SMC`。它从栈上取 `len`、`start`、256 位 `key`，将合约原始代码从 `start` 起按 32 字节密钥循环异或，随后把解出的 `len` 字节替换为当前执行字节码并从头执行。忽略这一点，用普通 EVM 跑合约会直接得到错误语义。

## 解题过程

### 动态追踪与 handler 提升

官方求解管线为：

```text
trace -> lift -> devirt -> emit -> deobf -> validate -> invert
```

先在补丁版 `revm` 中用固定明文 `00112233445566778899aabbccddeeff` 跑一遍合约。`insttrace` 记录每条 VM 指令的操作数、寄存器和内存状态，`mapdispatch` 则恢复 dispatcher 到 handler 的映射；追踪器也保存经 `0x0c` 动态解密后的 handler 代码和本次密文。

随后把每个实际出现的 handler 静态提升为使用 `i256` 的 LLVM 函数。数据操作数保持符号化，并对 handler 的输入输出做语义分类。这样既利用动态轨迹缩小了范围，也没有把那次固定输入硬编码进恢复结果。

### 去虚拟化与去混淆

从 EVM 返回值向后切片，按 SSA 依赖重建闭合的数据流图。管线采用 fail-closed 策略：任何 handler 语义未解析、依赖未闭合或重放密文与 EVM 观测值不一致都会停止，而不是猜测缺失节点。

确认重放逐位一致后，将图序列化为直线型 LLVM 函数 `@cipher(cd, out)`。轨迹中具体的 S-box 内存地址会被替换为恢复出的索引表达式，因此生成物是适用于任意 16 字节输入的加密函数，不是对一条轨迹的录制回放。

混淆层包含巨大诱饵常量池和成对抵消的数据流。求解器先验证把诱饵池清零不会改变输出，再运行 `opt -O3` 和迭代 `instcombine`。原本约 60 万行的 IR 会缩减到数万行，留下可辨认的密码结构。这里的验证顺序很重要：不能因为某个常量“看起来没用”就直接删掉。

### 恢复级联并逆向

从精简 IR 构造哈希合并、XOR 归一化的表达式 DAG，然后按 128 位级联边界从左到右识别各层：

- **AES-128：** 枚举第一轮某个 S-box 输入的 256 种取值，读出有效 S-box；验证其为“有限域求逆加仿射变换”，并恢复约简多项式、掩码和 11 组轮密钥。ShiftRows 与 MixColumns 再按 FIPS-197 结构确认。
- **双路 DES：** 从两个 64 位 lane 中的 $E(R)\oplus K_i$ 节点恢复各自 16 轮的 48 位子密钥，并核对 S-box 与置换结构。
- **SPECK-128/128：** 识别加、循环移位、异或组成的 ARX 轮，恢复旋转参数及 32 轮轮密钥。

算法顺序不是固定假设，而是根据每份二进制的阶段边界自动识别。对目标密文逆序执行相应解密后，四个明文块为：

```text
SEKAI{32a1544c39
91ef4ad7124990eb
bda67c0f19c9bb63
e0e9587f03a0b02}
```

每个结果都要通过四项检查：恢复模型重新加密命中目标、反混淆后的本地 oracle 命中目标、真实 EVM 合约命中目标，以及算法模型自身测试。打包的 runner 还把四块拼成 64 字节，以
$K_0=\operatorname{Keccak256}(P_1\Vert P_2\Vert P_3\Vert P_4)$、
$K_1=\operatorname{Keccak256}(K_0)$ 解开内置 `FLAG_BLOB`，从而避免在验证程序中直接保存明文。

最终得到：

```text
SEKAI{32a1544c3991ef4ad7124990ebbda67c0f19c9bb63e0e9587f03a0b02}
```

## 方法总结

本题的可靠性来自多层验证，而不是一次反编译后“看起来像某种密码”。面对超大 VM 保护程序，可以用动态轨迹限定真实执行面，用静态符号提升恢复输入无关的语义，再让编译器消除已经证明无效的垃圾。最终必须用随机输入比较 LLVM 与真实 EVM，并用目标明文正向重加密；两类检查分别防止单轨迹过拟合和参数识别错误。
