# UMDCTF 2025 - cmsc351

## 题目简述

附件是一个包含约 1000 个结构高度相似函数的 ELF。程序读取最多 24 个字符，从
`f0(285, 199, input)` 开始调用；绝大多数 `fN` 根据当前字符 `t` 或 `s` 跳到另一个
函数，只有目标函数 `f882` 直接检查两个 fuel 参数之和：

```c
bool f882(int fuel1, int fuel2, char *choices) {
    return fuel1 + fuel2 <= 0;
}
```

本质上，函数就是图节点，`t`/`s` 分支就是两条有向边。需要恢复一条从节点 0 到
节点 882、且最终 fuel 检查通过的路径。

## 解题过程

任取一个普通节点即可看出统一模板：

```c
bool f0(int fuel1, int fuel2, char *choices) {
    switch (*choices) {
    case 't':
        return f483(fuel2 - 2, fuel2 - 4, choices + 1);
    case 's':
        return f574(fuel2 - 3, fuel2 - 6, choices + 1);
    default:
        return false;
    }
}
```

因此逆向时无需逐个理解 1000 个函数的机器码，只要提取每个函数两条调用边、对应的
输入字符和两个减数。仓库中的生成器进一步确认：图由
`networkx.random_k_out_graph(1000, 2, 50, seed=66)` 生成，每条边权为两个减数
之和；生成时从 0 到 882 求了最短路。

恢复出的节点路径为：

```text
0 -> 483 -> 881 -> 455 -> 606 -> 167 -> 333 -> 758
  -> 980 -> 586 -> 120 -> 160 -> 961 -> 314 -> 345
  -> 478 -> 970 -> 289 -> 473 -> 882
```

逐节点查看目标位于 `t` 还是 `s` 分支，得到：

```text
ttstttsssssttsssttt
```

还需验证 fuel，而不能只看是否到达 `f882`。程序每次都以旧 `fuel2` 为基准生成下一对
参数；沿上述路径传播后，最后从 `f473` 的 `t` 分支进入：

```c
return f882(fuel2 - 15, fuel2 - 20, choices + 1);
```

此时旧 `fuel2 = 7`，故传入 `f882` 的参数为 `-8` 和 `-13`，总和 `-21`，
终止条件成立。实际向附件输入该字符串后输出：

```text
UMDCTF{C4ll_Gr4ph5_ar3_st1ll_gr4ph5}
```

## 方法总结

面对大量同构函数，不应逐函数线性阅读，而应先识别重复模板并抽象为图。节点由函数
地址或编号表示，边记录分支字符、目标函数和状态更新；之后用 BFS、Dijkstra 或带
状态的搜索寻找终点，并单独模拟路径上的参数变化。本题的 flag 也点明了核心：
调用图仍然是图，规模只会影响提取方式，不会改变问题结构。
