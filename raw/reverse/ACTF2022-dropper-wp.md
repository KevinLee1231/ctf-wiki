# dropper

## 题目简述

题目给出一个 64 位 Windows PE，外层程序经过 UPX 压缩，并以 dropper 形式把真正的校验程序藏在资源数据中。官方题解列出的分析链为：脱 UPX、定位资源提取与异或解密逻辑、获得内层 PE，再绕过异常处理与虚函数表跳转造成的静态分析干扰，最终还原大整数运算和 Base64 解码过程。

虽然样本采用了 dropper 技术，但没有以恶意行为、持久化或 C2 为解题目标；flag 取决于恢复内层程序的校验逻辑，因此归入 Reverse 更合适。

对这类未知 PE 应只在带快照、无共享目录和受控网络的隔离虚拟机中动态调试。原始题解中“把缺失 DLL 下载到 `System32`”的做法风险很高，也会污染系统环境；若确需补依赖，应核验来源，并优先放在样本的隔离目录中。

## 解题过程

### 1. 从外层 dropper 提取内层 PE

先根据 UPX 节名、入口桩和导入特征识别外壳，使用与样本匹配的 UPX 版本脱壳。脱壳后的主程序仍不是最终校验逻辑：它从 PE 资源区读取一段数据，经过逐字节异或还原成第二个可执行文件，再加载或创建子进程运行。

静态分析时如果找不到常见进程创建或 PE 加载函数的直接交叉引用，应继续检查 `GetProcAddress` 的调用点。外层程序动态解析 API，因而普通导入表和直接 xref 不会完整暴露加载路径。沿着“查找资源、加载资源、取得资源指针、异或循环、动态解析 API”的数据流，可以在内层数据恢复为 `MZ`/`PE` 结构后将其保存为独立文件。

也可以在隔离环境中动态定位：让外层运行到出现 `flag:` 提示，暂停父进程，再附加到它创建的子进程。这样可以绕过手工复现 dropper 加载器的工作，直接在内层映像的代码段设置执行断点。提取出的内层 PE 仍应保留哈希并与内存映像核对，避免把错误偏移或尚未完全解密的数据当成分析对象。

### 2. 找到被异常处理隐藏的校验路径

内层程序用一次除零异常把正常控制流转入异常处理器，并在该路径上修改或挂接虚函数表项。结果是反编译器展示的普通调用图与实际执行路径不一致，单靠静态交叉引用容易停在伪逻辑上。

可采用两种互相验证的定位方式：

1. 输入任意候选 flag，在内层 `.text` 区设置执行断点，异常触发后单步跟踪处理器返回位置和间接调用目标；
2. 在字符串窗口搜索 Base64 字母表，对引用位置回溯，定位把大整数转换为字符并进行最终比较的函数。

调试时记录异常前后的虚表指针、间接调用目标和大整数对象地址。大整数在内存中以多精度 limb 保存，调试器通常按十六进制展示；应按对象布局还原为一个整数，而不是把屏幕上的十六进制文本直接当作待比较字符串。

### 3. 重放大整数运算并解码

到达真实校验函数后，可以从运算对象中抄出初始大整数和各个常量。按程序顺序重放加、减、整除，得到一个以 128 为基数打包的整数 $N$。若编码字符串各字节为 $s_i$，其布局等价于

$$
N=\sum_{i=0}^{k-1}s_i128^i.
$$

所以循环取 $N\bmod 128$ 再令 $N\leftarrow\lfloor N/128\rfloor$，得到的就是从低位到高位排列的原始字符串，不需要再次反转。该字符串本身是 Base64，再解码一次即可得到 flag。

## 方法总结

本题的关键不是某一个复杂算法，而是连续识别三层障碍：UPX 只隐藏外层入口；资源异或与动态 API 解析隐藏内层 PE；除零异常和虚函数表跳转再隐藏真正的校验函数。沿数据流逐层剥离后，最终验证只是可直接重放的大整数四则运算、低位优先的 base-128 拆分和 Base64 解码。

处理 dropper 类逆向题时，应优先建立“外层加载器、内层载荷、真实校验函数”的分层证据链。静态分析用于确定资源与 API 解析关系，动态分析用于确认异常后的实际控制流，最后再用独立脚本验证常量和编码方向；任何一步都不应依赖在宿主系统直接运行未知样本。
