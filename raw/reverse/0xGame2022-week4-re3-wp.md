# week4re3

## 题目简述

题目原本设计为一个 42 字符输入校验程序，但实现中在循环里反复对正在被修改的输入调用 `strlen()`，使循环边界随内容变化，最终附件泄露了几乎完整的正确输入。

## 解题过程

问题不在复杂算法，而在动态循环边界。程序每轮都会改变 `input` 的内容，同时仍使用类似下面的条件：

~~~c
for (i = 0; i < strlen(input); ++i) {
    /* 此处会继续修改 input */
}
~~~

`strlen()` 每次都会重新寻找第一个 `\0`。一旦循环体在更靠前的位置产生空字节，下一次判断得到的长度就会缩短，程序不再按预期处理全部 42 个字符。这一构建错误使校验数据直接暴露在附件中，只剩前缀附近少量字符需要补全。

进一步复核源码仓库中的 `re3.exe` 可以确认它并不是以 `MZ` 开头的 PE 文件，而是 43 字节的 ASCII 文本：42 字节 flag 后跟一个换行。即使只按原题附件分析，也可先根据固定格式确定前五个字符为 `flag{`，再枚举第六个字符；第六位为 `f` 时即可拼出完整结果：

~~~text
flag{ff6cf093-f357-428d-a642-16b10702bae7}
~~~

## 方法总结

循环条件中的 `strlen()` 并不是预先计算的常量；当循环体会修改同一缓冲区时，边界也会变化。逆向时如果程序行为与设计明显不符，应同时检查附件格式、编译产物和边界条件，不必强行寻找不存在的复杂校验算法。
