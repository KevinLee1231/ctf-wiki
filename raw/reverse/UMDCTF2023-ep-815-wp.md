# EP_815

## 题目简述

题目给出一个可以正常运行的 VeSP 机器码程序 `program.vsp`。其表面逻辑只是若干
`ADD`、`SUB` 和最终 `HLT`，但机器码与规范汇编结果在部分低位上不同。

VeSP 指令字为 16 位。对 `ADD`、`CMP` 和 `HLT` 这类不需要操作数的指令，模拟器只根据高位 opcode 决定行为，低 12 位不会影响执行。作者正是把 flag 的十六进制数据塞进了这些未使用位。

## 解题过程

### 对比规范机器码

例如规范的 `ADD` 可编码为：

```text
0000
```

题目文件对应位置却出现：

```text
0554
```

高 4 位仍为 `ADD` 的 opcode `0`，所以执行效果不变；低 3 个十六进制数字 `554` 是隐藏载荷。官方说明把它称作“低 3 bytes”，但准确说法是低 3 个十六进制数位，也就是 12 bit。

同理，最终 `HLT` 的规范字 `7000` 被改成 `757D`，低 12 位携带 `57D`。

### 从执行轨迹收集未使用位

按手册从地址 `002` 加载程序，并启用 verbose trace。筛出即将执行 `ADD` 或
`HLT` 时的 IR 值，取每个指令字末 3 个十六进制数位，按执行顺序连接：

```bash
./vesp1_1 | tee trace.log

grep -B 3 -E 'ADD|HLT' trace.log \
  | grep 'IR' \
  | awk '{print $6}' \
  | cut -c 2- \
  | tr -d '\n,' \
  | xxd -r -p
```

`SUB` 在 VeSP 中是调用 `ADD`、`CMP` 等基础动作的伪指令，因此 verbose trace 仍会暴露承载数据的底层指令。连接后的十六进制流从
`554D44435446...` 开始，即 ASCII 的 `UMDCTF...`。

最终结果为：

```text
UMDCTF{smuggl3d_n0m5}
```

## 方法总结

- 核心技巧：识别无操作数指令中的未使用低 12 位，把不会改变程序语义的指令填充还原为连续隐藏字节。
- 识别信号：机器码能正常执行，但同一 opcode 的低位在不同位置呈现高熵、与规范汇编结果不一致时，应检查 instruction padding covert channel。
- 复用要点：按指令执行顺序拼接十六进制数位，并保留跨指令边界；每条指令贡献 3 个 hex digit，不一定独立对齐到完整字节。
