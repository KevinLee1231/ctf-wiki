# UMDCTF2022 Pi Trojan Writeup

## 题目简述

附件 `kernel.img` 是给 Raspberry Pi 3 使用的 AArch64 裸机镜像，启动后要求输入密码。镜像没有 ELF 头和符号，并在运行时对认证代码做两层原地 XOR 解密。目标不是求 SHA-256 原像，而是还原自修改代码并提取内置的目标摘要；官方允许把该摘要包进 flag 格式提交。

## 解题过程

按 Raspberry Pi 3 裸机镜像的装载地址 `0x80000` 反汇编：

```bash
aarch64-linux-gnu-objdump \
  -D -b binary -m aarch64 \
  --adjust-vma=0x80000 kernel.img
```

入口完成异常级别和串口初始化后跳到 `0x82160`。该函数用 64 位常量：

```text
0xcefaaddecefaadde
```

逐个异或文件偏移 `0x1e70..0x2110`，即运行地址 `0x81e70..0x82110`。离线执行同样的 XOR 后，这一段变成合法的 AArch64 认证函数。

解密后的认证函数还会用位于 `0x82150` 的第二个常量：

```text
0xbebafecabebafeca
```

解密 `0x822c0..0x82ce0` 的哈希实现。随后程序读取最多 `0x100` 字节的密码，计算 32 字节 SHA-256，并逐字节与地址 `0x84010` 的常量比较。

镜像基址为 `0x80000`，所以 `0x84010` 对应文件偏移 `0x4010`。直接提取：

```bash
xxd -g 1 -s 0x4010 -l 32 kernel.img
```

得到：

```text
49 07 bd cd fe 6d 8c be 34 28 9d 29 49 67 15 7d
56 9b c7 24 de 8b 4b 3f 1a aa b2 06 58 09 0c fd
```

连成十六进制摘要：

```text
4907bdcdfe6d8cbe34289d294967157d569bc724de8b4b3f1aaab20658090cfd
```

因此可提交：

```text
UMDCTF{4907bdcdfe6d8cbe34289d294967157d569bc724de8b4b3f1aaab20658090cfd}
```

官方答案文件还给出了等价的明文形式：

```text
UMDCTF-{P0lyMorPh1c_C0D3_1s_b3Tt3r}
```

对该完整字符串计算 SHA-256，结果正是镜像中的 32 字节常量，证明两种答案指向同一个认证值。

## 方法总结

裸机镜像应先确定装载基址，再把运行地址换算成文件偏移。遇到大量看似非法指令时，不应立即认定架构错误；本题入口明确执行了写代码段和刷新指令缓存，说明后续代码是运行时解密的。两层 XOR 还原后，最终校验只是普通 SHA-256 比较，提取目标摘要即可，不需要尝试不可行的通用哈希逆向。
