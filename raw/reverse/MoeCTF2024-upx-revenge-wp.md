# upx-revenge

## 题目简述

附件仍是标准 UPX 壳，但 PE 节名被从常见的 `UPX0`、`UPX1` 改掉，导致 `upx -d` 的格式检查失败。恢复节名后即可正常脱壳；也可在程序自解压后动态定位 OEP 或直接搜索解压到内存的明文。

## 解题过程

用 PE 查看器或十六进制编辑器检查节表，可以看到本应为 `UPX0`、`UPX1` 的名称被改成类似 `vmp0`、`vmp1`。节名字段固定为八字节，直接把对应四字节名称改回：

```text
vmp0 -> UPX0
vmp1 -> UPX1
```

保存副本后执行：

```bash
upx -d upx-revenge.exe -o unpacked.exe
```

在脱壳后的 `unpacked.exe` 中搜索字符串或跟到 flag 校验即可得到：

```text
moectf{554ea35c-a1bb-4d8f-a323-bd697564bf27}
```

若不修改文件，也可在 x64dbg 中运行至 UPX stub 完成解压并跳向原始入口点，再从进程内存 dump 映像、修复导入表；本题的明文也会在该阶段出现在内存中。Cheat Engine 的字符串扫描只是利用同一事实，并非另一种加密解法。

## 方法总结

`upx -d` 失败不代表壳结构已经彻底魔改。先比较 PE 节表、入口 stub 和典型 UPX 特征；仅改节名的样本恢复名称即可交给官方解包器，动态 OEP/dump 则是格式修复失败时的通用后备方案。
