# d3rc4

## 题目简述

题目是 Linux 多进程/管道混淆的逆向。程序在 `_init_array` 做初始化和字符串解码，在 `_fini_array` 放关键校验；主体里用 pipe IPC 和多进程 prime sieve 制造大量干扰。真正需要恢复的是父进程最终校验中的 RC4/加密逻辑，子进程对全局变量的修改不会影响父进程最终结果。

## 解题过程

### 关键点

### _init_array 与 _fini_array

注册在 `_init_array` 和 `_fini_array` 中的函数会分别在 main 函数前后执行。

`sub_1A20`（位于 `_init_array`）负责部分变量初始化和字符串解码工作。

`sub_16C5`（位于 `_fini_array`）是关键函数。

### pipe 进程间通信

pipe 是 Linux 中的一种 IPC 方式，可以把数据单向地从一个进程传给另一个进程。

本题中，pipe 用于在父子进程之间建立通信通道，`read` 和 `write` 函数用于在通道上读写数据。

### 多进程质数筛

多进程筛法的思路来自 Russ Cox 的线程/并发示例。重要模式是一条进程管线：每个进程从父进程读取数字，把第一个数字作为 prime/base，过滤掉该 base 的倍数，再把剩余数字传给下一个子进程。在本题中，这个结构主要用于混淆，不是最终 flag 校验。

例如，初始时主进程获得数字 2 到 35。它会选择第一个数字 2 作为 `base`，然后对剩余数字逐个判断：如果 `n mod 2 == 0`，说明它不是质数并丢弃；否则把 `n` 传给子进程。子进程也会选择收到的第一个数字作为 `base`，这里是 3，然后对后续收到的数字做同样过滤：如果 `n mod 3 == 0` 就丢弃，否则传给下一层子进程。直到没有数字需要再传给子进程时，就得到了全部质数。

**一些干扰点**

由于父进程和子进程的内存空间完全隔离，子进程对全局变量的修改不会影响父进程。最终 check 在父进程中完成，所以子进程中的一些操作只是干扰，不会影响结果。

### 预期解

预期解是在理解算法的前提下还原加密逻辑，跳过 prime sieve 部分代码，最后暴力得到 flag，也可以尝试 z3 这类约束求解工具。

预期脚本只需要还原加密/check 逻辑，并跳过 prime-sieve 干扰。筛法流程用于解释 IPC 噪声，但最终验证仍然位于父进程。

## 方法总结

- 核心技巧：`_init_array`/`_fini_array` 入口识别、pipe IPC 去噪、多进程 prime sieve 干扰剥离、RC4/校验逻辑还原。
- 识别信号：逆向中出现大量 fork/pipe 且父子进程全局变量隔离时，要确认最终 check 在哪个进程发生，避免把子进程干扰路径当主逻辑。
- 复用要点：外部并发筛示例只解释进程管道结构；解题时应把它视作控制流噪声，优先保留父进程最终加密关系。
