# Xor-Endian

## 题目简述

附件是一个 64 位、x86-64、小端序、动态链接且未去符号的 PIE ELF。程序仍然使用循环异或，但密文以 32 位整数数组保存；解题时必须先按小端序把每个整数还原为 4 个原始字节。

## 解题过程

IDA 中可以看到由多个十进制整数构成的密文数组。C 语言中的 `int` 通常占 4 字节。在这份 x86-64 ELF 中，多字节整数按小端序存储，例如整数的最低有效字节位于最低地址。因此不能直接把十进制整数当作字符使用，而要对每项执行 `to_bytes(4, "little")`。

程序使用的循环密钥是：

```text
Key0xGame2024
```

完整还原脚本如下：

```python
data = [
    1363025275, 253370901, 1448151638, 1415391232, 91507463,
    139743552, 1450318164, 1985283101, 1465125718, 1934953223,
    84430593,
]
key = b"Key0xGame2024"

ciphertext = b"".join(value.to_bytes(4, "little") for value in data)
plaintext = bytes(
    value ^ key[index % len(key)]
    for index, value in enumerate(ciphertext)
)
print(plaintext.decode())
```

IDA 有时会把 4 字节常量显示成单引号包裹的多字符常量，界面中的书写顺序容易造成误读。重新反编译后可以看到更符合内存字节顺序的表示，但脚本中仍应明确指定 `little`，避免依赖界面显示方式。

脚本输出：

```text
0xGame{b38ad4c8-733d-4f8f-93d4-17f1e79a8d68}
```

## 方法总结

这道题的核心不是异或本身，而是密文的表示形式。遇到整数数组保存字节流时，要先确认整数宽度和目标架构端序，再恢复原始字节并应用循环密钥；否则每 4 个字节的顺序都会颠倒。
