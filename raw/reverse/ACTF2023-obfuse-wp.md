# Obfuse

## 题目简述

题目是一个静态链接的简易 Shell，但编译时叠加了四类 LLVM 混淆：间接跳转、间接调用、增强版 OLLVM 控制流混淆，以及 Pluto 的全局变量加密。Shell 在执行命令前进入 `EvalCommand`，其中 `save` 根据当前后台进程数量向内部字符串追加一个字符，`check` 则校验累计的 25 字节内容。

目标是先恢复可分析的控制流和运行期全局数据，再拆解 `check` 中互不相同的三段校验：4 字节 MD5、15 字节修改版 RC5/AES，以及 6 位十进制 Trie 路径校验。

## 解题过程

### 1. 按性质分别处理四层混淆

四类混淆不应使用同一种手段硬拆：

- **全局变量加密**：加密常量在程序启动时必须自行解密，否则业务代码无法使用。应在初始化结束后附加调试器，直接查看或转储已解密的全局区；继续静态猜密文没有收益。
- **间接调用**：跟踪函数指针或跳转表的定义，记录每个调用点可能到达的真实 callee，再把分析数据库中的调用边补回。
- **间接跳转**：以当前基本块末尾的寄存器状态为输入，计算实际跳转目标；目标模式稳定时可直接匹配并修补为直接分支。
- **增强 OLLVM**：分发状态与不透明条件会破坏反编译器 CFG。可用 Unicorn 从真实块入口模拟到下一次间接转移，记录目标并以 BFS 遍历未访问块；遇到依赖输入的条件则分别探索可行分支，最后依据边集合重建 CFG。

Binary Ninja IL、IDA 脚本或其他框架都能实现上述思路，工具不是解题结论。关键产物应是“真实基本块及其后继关系”，而不是一份只对某个地址有效的 NOP 脚本。

题目所用 [Pluto](https://github.com/bluesadi/Pluto) 是基于 LLVM 14.0.6 的编译期混淆器；仓库目前已停止维护，但源码仍能用于核对 pass 的设计。外链在本题中的重要信息是它在编译期把程序变成运行时等价的复杂形式，而全局数据必须在使用前恢复；WP 已写明这一行为，不依赖读者另行查阅。原文引用的 enhanced OLLVM 论坛页面目前访问不稳定，且解题所需的控制流恢复原则已在正文说明，因此不再保留该 URL。

### 2. 定位 `save` 与 `check`

等全局区在运行时解密后，搜索命令词 `save`、`check`，或对字符串比较函数下断点，即可定位 `EvalCommand` 的两个分支。

`save` 保存的不是命令参数，而是当前后台进程数量对应的字符。按原交互完成输入，需要通过启动和结束后台任务调节计数，再调用 `save` 逐字节追加；分析阶段更直接的做法是 patch `save`，让它从标准输入取字符，或在 `check` 前直接改写内部缓冲区。后者只绕过繁琐的 Shell 输入编码，不绕过三个校验本身。

`check` 先要求总长度为 25，然后切分为：

```text
s1 = input[0:4]
s2 = input[4:19]
s3 = input[19:25]
```

其中 `s3` 还必须全部为十进制数字。

### 3. 第一段：MD5

第一段满足：

```cpp
MD5(s1) == "2aedfa0f134e41fa06a0dd4f8c6fba80"
```

长度只有 4，可按可打印字符爆破；候选也可直接做一次 MD5 核验。结果是：

```text
s1 = W4@t
MD5(s1) = 2aedfa0f134e41fa06a0dd4f8c6fba80
```

### 4. 第二段：修改版 RC5 与 AES

程序先让修改版 RC5 处理 15 字节 `s2`，产生 16 个 16 位值，再按大端顺序拆成 32 字节作为修改版 AES 的输入：

```cpp
rc.Init(s2, rc5_enc);
for (int i = 0; i < 16; ++i) {
    aes_1[i * 2]     = rc5_enc[i] >> 8;
    aes_1[i * 2 + 1] = rc5_enc[i] & 0xff;
}
AES(aes_1) == aes_flag;
```

这里不能直接套标准库：RC5 修改了魔数 $P_w$、$Q_w$；AES 修改了 ShiftRows 顺序，而且 S-box 不是标准表，而是在运行时按下式生成：

```cpp
for (int i = 0; i < 256; ++i)
    sbox[i] = aes_8bit_mul_mod_0x101(aes_8bit_inverse(i), 0x1f) ^ 0x37;
```

可靠的恢复顺序是：

1. 在 `build_sbox()` 返回后转储 256 字节 S-box，并建立逆表；
2. 从解密后的代码确认修改版 ShiftRows 置换及轮数，逆向两块 AES 输出；
3. 将得到的 32 字节重新组合为 16 个 RC5 字，恢复被修改的 $P_w$、$Q_w$ 和轮密钥；
4. 逆 RC5 得到 15 字节明文，再正向执行两层算法核对 `aes_flag` 的全部 32 字节。

公开仓库没有给出 RC5/AES 的完整实现，不能只凭常量伪造一份可运行解密器；但官方局部源码中的注释给出了经上述流程验证的结果：

```text
s2 = _0_N41v3_$#4ll_
```

### 5. 第三段：确定性十叉树

第三段先固定随机种子，再以深度优先顺序构造深度为 6 的十叉树：

```cpp
srand(114514);

void gentree(int node, int depth) {
    val[node] = rand();
    if (depth >= 6) return;
    for (int digit = 0; digit <= 9; ++digit) {
        edge[node].push_back(++cnt_tree);
        gentree(cnt_tree, depth + 1);
    }
}
```

每一位数字选择当前节点的第 `digit` 个孩子，并累计：

```cpp
pos = edge[pos][digit];
ans ^= val[pos] * pos;
```

最终要求 `ans == -1175156229`。种子固定使所有节点编号和值可复现；总节点数为 $1+10+10^2+\cdots+10^6=1111111$，候选只有 $10^6$ 个。用与附件一致的 glibc `rand()` 建树，按 32 位补码语义计算乘法和异或，再枚举 `000000` 至 `999999`，得到：

```text
s3 = 120911
```

不能使用 Windows CRT 的 `rand()` 代替 glibc 实现，否则相同种子会产生不同树。

### 6. 拼接并验证

三段连接后，程序接受的 25 字节字符串是：

```text
W4@t_0_N41v3_$#4ll_120911
```

将它写入内部缓冲区并执行 `check`，三个分支均通过，程序进入 `Correct Flag` 路径。仓库没有提供额外的 `ACTF{...}` 包装规则，因此该值应表述为程序的正确待校验串，而不是擅自添加前后缀。

## 方法总结

本题不是“看到 OLLVM 就跑某个脚本”，而是分层恢复语义：运行后提取全局明文，模拟间接控制流并重建 CFG，再从 `EvalCommand` 进入真实校验。三段数据分别用小空间哈希爆破、定制密码算法逆运算和固定 PRNG 的树路径枚举解决。

复现时最容易出错的是把标准 RC5/AES 当成题目实现、忽略 32 位整数回绕，或在非 glibc 平台复现 `rand()`。每段都应进行正向校验，最后再验证完整 25 字节输入，而不是仅依赖源码注释拼接答案。
