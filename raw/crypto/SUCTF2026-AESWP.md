# SUCTF2026-AES

## 题目简述

题目给出一个可交互修改 S-box 的 AES-128 ECB 服务。服务启动时随机生成 128 位 `seed` 和 `key`，用 `Random(seed).shuffle` 打乱标准 AES S-box，再用该 S-box 展开轮密钥并输出 PKCS#7 填充后的 flag 密文。菜单允许：

- `change(seed, key)`：重新采样当前 S-box，或重新展开轮密钥；
- `encrypt(message)`：加密任意消息；
- `reset`：恢复最初的 secret seed、key 和轮密钥状态。

核心漏洞位于：

```python
def change(self, s=None, k=None):
    if s:
        self.Sbox = Random(s).choices(self.Sbox, k=256)
    if k:
        self.change_key(k)
```

`choices` 是有放回采样，不再保持 S-box 为置换；而且只传 seed 时，数据通道使用新 S-box，轮密钥却仍由旧 S-box 和旧 key 生成。这两个性质分别提供了“可控塌缩”和“保留 secret 轮密钥”的攻击面。

经源码核对，`chal.py` 实际执行 `range(300)`，即最多 300 次菜单操作。官方 WP 中“310 次”的预算和 `exp.py` 里的 `MAX_MENU=310` 是旧参数残留；仓库当前官方利用采用 `SWITCH=215`，基础流程实际消耗 299 次，能够落在真实限制内。

## 解题过程

### 1. 把 `Random(seed).choices` 看成索引映射

设重置后的未知 S-box 为置换 $S_0$。对固定 seed，本地可确定采样索引序列

$$
f_s(i)=\operatorname{Random}(s).choices(\{0,\ldots,255\},256)[i],
$$

调用一次后有

$$
S_1(i)=S_0(f_s(i)).
$$

连续调用时则不断复合这些索引映射。攻击者虽然不知道 $S_0$ 的值，但可以精确设计或搜索 $f_s$ 的结构。

### 2. 官方解法：构造采样索引并恢复初始 S-box

官方解法使用 `recover_seed.py` 反求能够产生指定 `choices` 索引序列的 seed，并设计两类模板：

```text
single(t) = [t] * (t + 1) + [t + 1, ..., 255]
pair(t)   = [t] * (t + 1) + [t + 1] + [t + 2, ..., 255]
```

它们分别把前缀压成一个值，或保留前缀边界的两个候选值，同时让尚未恢复的尾部尽量保持原状。每次采样还提交已知 key，使轮密钥在当前 S-box 下重新展开，从而把对应密文转化为对 $S_0[t]$、$S_0[t+1]$ 的离线约束。

仓库当前 `exp.py` 的在线采集参数为：

```text
SWITCH = 215
EXTRA_SINGLE = 1
pair:   t = 2, 4, ..., 214        共 107 组
single: t = 215, 216, ..., 255    共 41 组
```

每组使用一次 `change` 和一次 `encrypt`，最后提取第 10 轮密钥还需 `reset + change + encrypt`，总菜单操作数为

$$
2\times(107+41)+3=299.
$$

离线阶段由 `hybrid_recover.go` 完成：

1. 从 `single(255)` 向低位恢复尾部，因为高位模板包含的未知量最少；
2. 对低位的 `pair` 约束做 DFS 回溯；
3. 先用一个 16 字节块快速筛候选，再用完整密文确认，并用多线程、随机重启处理偶发歧义。

该阶段恢复 $S_0[2],\ldots,S_0[255]$。由于 $S_0$ 原本是 0–255 的置换，索引 0、1 对应的两个剩余值只需枚举两种排列。

### 3. 用常值 S-box 提取 `K10`

恢复 $S_0[2]$ 后，先 `reset`，再构造索引序列 `[2] * 256` 对应的 seed，并只修改 S-box、不修改 key。当前 S-box 因而成为常值函数：

$$
S(x)=S_0(2)=v_2.
$$

每轮 `SubBytes` 都把状态压成同一个字节；全相等列经过 `ShiftRows` 和 `MixColumns` 后仍保持全相等。最后一轮没有 `MixColumns`，所以任意单块密文满足

$$
C=[v_2]^{16}\oplus K_{10}.
$$

因此

```python
K10 = xor(ciphertext_block, bytes([S0[2]]) * 16)
```

即可得到 secret 状态下的最后一轮密钥。

### 4. 逆密钥扩展并解密

已知完整候选 S-box 和 $K_{10}$ 后，AES-128 密钥扩展是可逆的。从 `w[43]` 逆推至 `w[0]`；在 `i % 4 == 0` 的位置使用恢复出的自定义 S-box 和对应 `Rcon`。对 $S_0[0],S_0[1]$ 的两种排列分别：

1. 逆推 16 字节 master key；
2. 用该 S-box 和 key 实现逆 AES；
3. 解密初始 flag 密文并做 PKCS#7 去填充。

能得到合法明文的一组即为正确排列，结果为：

```text
SUCTF{Th3_K3y_15_1n_7h3_S_80x}
```

### 5. 总 PDF 中的低交互替代路线

参赛 WP 利用了同一漏洞，但不反求精确索引序列。它离线找到 `seed=138188`，连续调用 18 次后，复合映射 $f_s^{18}$ 的像集只剩一个索引，因而也能把未知 S-box 压成常值。

常值本身未知时，可在塌缩状态下把 key 改为已知值 1，查询一个块，再本地枚举 256 个常值并模拟 AES，确定实际常值。随后 `reset`、再次塌缩但保留 secret 轮密钥，即可按上一节公式恢复 $K_{10}$。

恢复原始 S-box 时，对若干 probe seed 分别执行一次采样。对 seed $s_i$，本地已知索引像集

$$
I_i=\operatorname{Im}(f_{s_i}),
$$

去掉密文末轮密钥并逆 `ShiftRows` 后，大量随机明文会暴露当前 S-box 的值域

$$
V_i=S_0(I_i).
$$

于是索引 $x$ 和值 $S_0(x)$ 具有同一成员指纹：

$$
(x\in I_1,\ldots,x\in I_n)
=(S_0(x)\in V_1,\ldots,S_0(x)\in V_n).
$$

总 PDF 选择了 12 个能让 256 个索引指纹互异的 seed，从而一一匹配并重建整个 $S_0$。这条路线交互次数更少，但需要每个 probe 收集足够多的随机块，直到完整观察相应值域。

实现交互时不要给可选整数发送空行。源码表达式 `int(input(...) or 0, 16)` 在空输入时会变成 `int(0, 16)` 并触发 `TypeError`；发送字符串 `"0"` 才会稳定解析成 `None`。

## 方法总结

- 漏洞不是 AES 本身，而是 S-box 更新与轮密钥更新不同步，以及 `choices` 有放回采样破坏了置换性质。
- 官方路线精确构造 `choices` 的索引结构，在 299 次真实菜单预算内恢复 S-box；常值 S-box 再把所有早期轮信息抹掉，只留下 $K_{10}$ 的异或掩码。
- 总 PDF 路线用复合映射塌缩和集合成员指纹恢复置换，交互更少，展示了“不必恢复完整映射，只需控制像集”的另一种思路。
- 审题时必须以实际源码的循环次数和当前利用参数为准，不能直接照搬 WP 中已经漂移的预算常量。
