# GoShop

## 题目简述

题目实现了一个 Go 商店。用户初始余额为 `100`，商品 `Apple` 的单价为 `10`，而购买 `Flag` 需要 `999999999`。购买接口只拒绝负数数量，却没有限制数量上界；后续又在 `int64` 中计算总价，因此可以利用有符号整数溢出把总价变成负数，反向增加余额。

## 解题过程

购买逻辑的关键部分如下：

```go
n, _ := strconv.Atoi(data["num"].(string))

if n < 0 {
	return
}

if user.Money >= product.Price*int64(n) {
	user.Money -= product.Price * int64(n)
	user.Items[product.Name] += int64(n)
}
```

`n < 0` 只能阻止直接提交负数。`product.Price * int64(n)` 的结果仍是 `int64`；当数学结果超过 $2^{63}-1$ 时，它会按 64 位整数回绕。选择购买

```text
Apple × 922337203695477808
```

其数学总价为

$$
10 \times 922337203695477808
= 9223372036954778080.
$$

该值按 64 位有符号整数解释时等于

$$
9223372036954778080-2^{64}
=-9223372036754773536.
$$

于是余额判断变成 `100 >= 负数`，扣款操作则实际执行

$$
100-(-9223372036754773536)
=9223372036754773636,
$$

余额被增加到足以购买 `Flag`。拦截购买请求，将 JSON 中的商品名设为 `Apple`、数量字符串设为上述数值即可。

页面返回的购买结果为：

```text
Buy Apple * 922337203695477808 success
```

随后按正常流程购买一个 `Flag`，再访问取 flag 的接口。

```text
0xGame{eaf905e4-28a1-4006-b8a0-e8ddc2d673bf}
```

## 方法总结

本题的漏洞不是“允许负数”，而是只校验了输入的符号、没有校验乘法结果是否越界。审计金额、长度和计数逻辑时，应同时检查输入范围与中间运算；服务端可在乘法前验证 `n <= math.MaxInt64/price`，并对转换错误进行处理。
