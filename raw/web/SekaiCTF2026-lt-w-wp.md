# \<\w+

## 题目简述

应用允许创建和更新笔记，读取时又把文件内容直接作为 `text/html` 返回。输入先经过 `bluemonday.StrictPolicy()`，随后程序反向把 `&lt;`、`&gt;` 还原为尖括号，最后删除匹配 `<(/)?\w+` 的片段。单个请求很难留下完整标签，但更新接口对同一文件执行无锁的 `O_TRUNC` 打开和覆盖写入，破坏了“每个落盘文件都来自一次已过滤请求”的假设。

## 解题过程

### 构造两个分别合法的写入

目标 payload 为：

```html
<img src=x onerror=alert(document.cookie)>
```

将它拆成两个更新内容：

```text
p1 = <
p2 = *img src=x onerror=alert(document.cookie)>
```

`p1` 只有一个孤立 `<`，不会匹配要求后面紧跟单词字符的正则；`p2` 没有起始 `<`，也只是普通文本。`*` 是第一个字节的占位符，后续要由另一个文件描述符覆盖。

### 利用独立文件偏移覆盖首字节

两个 PUT 都执行：

```go
os.OpenFile(filePath, os.O_WRONLY|os.O_TRUNC, 0644)
f.Write([]byte(sanitized))
```

关键并不是一个请求从“后续偏移”写剩余内容，而是两个独立文件描述符的当前偏移都从 0 开始。命中的调度顺序为：

1. 写 `p1` 的请求先打开文件并完成截断，但暂未写入；
2. 写 `p2` 的请求再打开并截断，然后从偏移 0 写入完整的 `*img ...>`；
3. 第一个请求仍持有自己的文件描述符，最后从偏移 0 写入一个字节 `<`。

第三步只覆盖 `p2` 的首字节，不会再次截断，也不会缩短现有文件，于是磁盘内容恰好变为完整 `<img ...>`。这个最终字节串从未作为整体经过 sanitizer。

官方 Go solver 循环并发发送两个 PUT，再 GET 检查结果：

```go
go updateNote(id, p2)
go updateNote(id, p1)
```

当返回正文与完整 payload 相同时停止，将该笔记 ID 提交给管理员 bot。bot 打开笔记后触发 `onerror`，攻击脚本读取 flag。

外层 Nginx 关闭了请求和响应缓冲，并把请求转发给同一个 Go 实例，使竞争可稳定重复；创建接口有限速，但 PUT 没有限速。bot 会把 flag 放入当前站点的 `FLAG` Cookie，且比赛环境禁止访问外网，因此应通过题目提供的浏览器 console/dialog 输出 `document.cookie`，而不是向外部域名发请求。

最终得到：

```text
SEKAI{l0g1c_l1v3s_1n_c0d3..._vuln_l1v3s_1n_t1m3!}
```

## 方法总结

过滤器对两个输入分别成立，但最终文件不是任何一次过滤结果，而是两个独立文件描述符对同一 inode 的竞态覆盖。安全属性必须覆盖从验证到持久化的完整事务；更新时应对每个 note 串行化，或将过滤结果写入唯一临时文件后原子替换。仅给单次 `Write` 加锁而把 `O_TRUNC` 留在锁外，同样不能修复这个竞态。
