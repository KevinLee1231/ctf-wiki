# offknock

## 题目简述

DNS 解析器不按语义比较查询名，而是在原始 UDP 报文的字节区间 `[12:30]` 中搜索：

```text
\x04flag\x06market\x03polȳ
```

`polȳ` 的 UTF-8 编码占 5 字节，却被前缀 `\x03` 声明为 3 字节 label。普通 DNS 库会把余下两个字节解释为新的标签或压缩指针并报错，因此需要手工构造一份既包含目标原始字节、又能被 dnslib 完整解析的畸形报文。

## 解题过程

先构造 DNS 头，令 `QDCOUNT=1`、`ANCOUNT=1`。问题名开头为：

```python
poly = "polȳ".encode("utf-8")
qname = b"\x04flag\x06market\x03" + poly
```

UTF-8 字符 `ȳ` 编码为 `c8 b3`。在解析完声明的 3 字节 `pol` 后，dnslib 会把 `c8 b3` 当作 DNS 压缩指针；去掉最高两位后的目标偏移是：

$$
0xC8B3\ \&\ 0x3fff=0x08B3=2227.
$$

为让该指针落在报文内部且指向零字节，官方解法追加一条超长 TXT answer：RDATA 长度为 2183，内容全为零。这样总报文覆盖偏移 2227，压缩指针最终解析到根标签，整份报文可以通过 dnslib。

解析器随后仍在原始字节中找到预期片段，于是返回：

```text
UMDCTF{175_4_107_34513r_70_M4K3_KN0CK0FF5_N0W}
```

## 方法总结

漏洞来自“原始字节检查”和“协议语义解析”使用了不同视图。畸形 UTF-8 尾字节可被重新解释为 DNS 压缩指针，再用可控的长 answer 为指针准备合法落点。安全代码应先完成严格解析，再对规范化后的域名标签做比较，不能在固定字节窗口中搜索。
