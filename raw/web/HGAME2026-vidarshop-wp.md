# Vidarshop

## 题目简述

题面是商店服务：商品价格异常高，管理员能管理所有人的钱，且题面暗示用户名和 uid 之间存在关系。这题只有在线靶机，没有附件源码。根据现有题解记录，服务中真正参与身份识别的是 `uid` 而不是看似显眼的 JWT 头；用户注册行为能暴露用户名字符和 uid 数字之间的对应关系，管理余额的接口还存在对象属性更新逻辑。

## 解题过程

这题的第一层干扰是 JWT 请求头。实际测试后可以发现服务并不依赖这个头判断身份，真正关键的是 `uid`。多注册几个账号并观察返回数据，可以建立用户名字符和 uid 数字之间的映射关系；按这个映射可以构造出对应 `admin` 的 uid，例如题解中记录的 `1413914`，也可以直接注册能映射到管理员 uid 的账号。

拿到管理员身份后，再看余额更新接口。`/api/update` 存在对象属性更新逻辑，能通过类似 `__proto__`/constructor 的入口污染对象原型；而商品购买逻辑读取的 `balance` 又受这个全局属性影响。于是利用链是：先伪造/获得管理员 uid，再向更新接口提交污染 payload，把余额或相关全局状态改成可购买高价商品的值，最后触发购买拿到 flag。

## 方法总结

遇到商店、余额、用户资料更新类题，先区分真正参与鉴权的字段和干扰字段。若身份字段可由用户名可预测映射得到，应先伪造高权限身份，再检查对象合并、属性更新和 `__proto__`/constructor 入口是否能污染全局状态。
