# You Want Me To Run What??

## 题目简述

服务接收一段 $10$ 到 $16$ 字节的合法 UTF-8 文本，把原始字节映射到 RWX 内存并直接执行。只有代码能够正常返回，且该字节串的 MD5 十六进制结果同时以 `000` 开头、以 `000` 结尾时，服务才返回 flag。

## 解题过程

输入必须同时满足三类约束：

1. 截断到换行或 NUL 前的长度在 $[10,16]$；
2. 整个缓冲区通过 UTF-8 校验，执行时不触发捕获的崩溃信号；
3. `md5(input).hexdigest()` 的前三位和后三位均为 `000`。

在 x86/x86-64 中，单字节 `0xc3` 是 `ret`。合法 UTF-8 不能让 `0xc3` 单独出现，但字符 `Ã` 的 UTF-8 编码正好以它开头：

```text
字符：Ã
UTF-8：c3 83
```

CPU 从缓冲区首字节执行 `0xc3` 后立即返回，后面的 `0x83` 与其余文本不会执行；UTF-8 解码器却把前两个字节视为一个合法字符。于是只需暴力搜索后续可打印字符，使整体 MD5 满足 $24+24=48$ 位的前后零约束。

仓库给出的一个有效输入是：

```text
ÃIPGwZkdRIE
```

其 UTF-8 字节、长度和 MD5 为：

```text
c3 83 49 50 47 77 5a 6b 64 52 49 45
12
000b9aba1e0bb233f90472fef0680000
```

发送该字符串并附加换行，执行阶段由首字节 `ret` 安全返回，哈希检查也通过，最终得到：

```text
UMDCTF{cpus_love_collisions}
```

这里并不是寻找 MD5 碰撞，而是在约 $2^{48}$ 的条件空间内做前缀与后缀原像搜索；题目名称中的 “collisions” 只是宽泛表述。

## 方法总结

- 核心技巧是区分“UTF-8 校验看到的字符序列”和“CPU 看到的机器码字节”。
- 让首条指令立即 `ret`，可以避开后续随机字符被执行的问题。
- 哈希条件是截断摘要匹配，不需要构造两条消息具有相同 MD5。
- 发送时必须保留 `Ã` 的 UTF-8 编码；终端若把它错误转码，首字节和哈希都会改变。
