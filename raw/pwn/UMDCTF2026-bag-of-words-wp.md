# bag_of_words

## 题目简述

题目实现一个 Scrabble 风格游戏。交换字母时，程序只拒绝 `n_tiles == 0`，却没有检查请求交换的字母数是否超过当前字母袋中的数量。计数使用无符号整数，使越界读取与下溢共同产生堆外/栈外信息泄露和越界写。

仓库中的 `interact.py` 是漏洞验证脚本，但包含 GDB 辅助跳过单词检查和修正一处泄露半字节，因此它不是完全独立的远程一键解法。下面区分可确认原语与本地 PoC 的辅助部分。

## 解题过程

当 $0<n_\text{tiles}<n_\text{letters}$ 时，代码从

```text
tiles + n_tiles - n_letters
```

开始读取，地址落在数组之前；随后无符号减法把剩余数量变成很大的值。程序把交换出的字母随机放回时，又以这个下溢后的范围计算位置，形成越界写。

随机数由 `srand(time(NULL))` 初始化。根据开局已排序的字母架，可以枚举相邻秒级时间种子，重放 glibc `rand()` 和 Fisher-Yates 过程，恢复后续随机位置。官方 PoC 利用这一点组织多轮交换：

1. 通过越界读让相邻内存字节进入字母架，逐步泄露栈和 libc 数据；
2. 由泄露地址计算 libc 基址；
3. 重放 PRNG，预测每次越界写落点；
4. 覆盖保存的 `rbp`（PoC 中偏移 `0x90`）和保存的 `rip`（偏移 `0x98`）；
5. 把栈迁移到可写区域，并跳转到与目标 libc 匹配的 one-gadget。

需要注意，随仓库提供的 PoC 用 GDB 强制绕过 `check_word`，还用 `leak_cheat` 补了一个不稳定的半字节，成功率标注约为 1%。因此远程化时必须自行解决合法单词路径、增加泄露冗余，并对模糊半字节枚举验证，不能把本地调试器行为当成服务端能力。

成功取得 shell 后 flag 为：

```text
UMDCTF{unfortunately_nigel_richards_knew_the_strategy_too}
```

## 方法总结

漏洞根源是“数量关系未验证 + 无符号下溢 + 可预测 PRNG”。它先把数组边界外的数据送回界面，再把攻击者可预测的随机索引变成写原语。官方仓库只给出了带调试器辅助的 PoC，复现时应明确这一证据边界，而不是声称脚本原样即可稳定攻击远端。
