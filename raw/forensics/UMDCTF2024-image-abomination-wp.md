# image abomination

## 题目简述

题目给出一张 2048×1024 的 JPEG。普通解码器仍能打开它，但只恢复出顶部一条橙色图像，其余区域变成大块灰色，并报告 `premature end of data segment`。这说明文件不是简单缺少 JPEG 文件头，而是压缩码流中的错误破坏了解码同步；常规十六进制补头无法重建后续 DCT 块。

![损坏 JPEG 仅解码出顶部橙色条带，其余区域被灰色填充](UMDCTF2024-image-abomination-wp/jpeg-decoding-failure.jpg)

README 指向 Two-ACIR。该框架专门恢复“加密 JPEG 码流解密后仍带有比特错误”的图像，任务核心是损坏证据恢复，因此归入 Forensics。

## 解题过程

### 为什么普通修图方法不够

JPEG 熵编码具有上下文和变长码特性。码流中少量比特翻转可能让后续 Huffman 符号边界整体错位，导致大片 MCU 无法解释。当前图片能显示顶部区域，说明 SOI、量化表、尺寸等结构仍在；问题集中在扫描数据，而不是扩展名或文件头。

因此应保留原始文件，避免让图片编辑器“另存为”后丢失损坏位置和内嵌低分辨率线索。

### Two-ACIR 两阶段恢复

[Two-ACIR 官方仓库](https://github.com/wenyang001/Two-ACIR/) 对这类错误采用以下链路：

1. **鲁棒 JPEG 解码**：尽可能完整解释损坏码流，同时提取低分辨率缩略图。
2. **SCA 阶段**：估计颜色偏移和块位置偏移，按块完成颜色补偿与对齐。
3. **GCA 阶段**：把缩略图作为全局引导，先用 coarse-guided pix2pix 恢复粗结构，再以拉普拉斯金字塔融合细化高分辨率像素。

也就是说，外链的重要价值不是“有个 AI 修图工具”，而是提供了与本题损坏模型匹配的鲁棒解码、块级校正和缩略图引导三部分。

### 执行恢复

按上游仓库准备其依赖、模型权重和目录结构，把原始 `flag.jpg` 放入待恢复数据目录。先运行鲁棒解码与第一阶段：

```bash
python decoder.py
```

上游会把块级恢复结果写入 `Input/` 和 `Input_S/`。再针对 2:1 的高分辨率场景运行第二阶段；仓库给出的 1K 配置形式为：

```bash
python test.py \
  --name Cityscape1K \
  --resize_or_crop scale_width \
  --scen 2 \
  --dataroot ../Datasets/test1K \
  --gpu_ids 0 \
  --which_epoch 200
```

实际目录和 GPU 编号按本地环境调整，但不能省略鲁棒解码产生的输入与缩略图。恢复后的画面中可读出：

```text
UMDCTF{mY4Rrr4K15myDuN3!!1}
```

## 方法总结

- 核心技巧：针对 JPEG 熵编码失步，先鲁棒解码并做块级颜色/位置校正，再利用保留下来的缩略图引导像素级恢复。
- 识别信号：JPEG 头和尺寸正常、图像只在前部可见、后续大面积灰块且解码器提示数据段提前结束时，应怀疑扫描码流比特错误。
- 复用要点：不要把能打开误认为文件健康，也不要先重编码；神经恢复前的鲁棒码流解码和块对齐是本题不可替代的步骤。
