# 加密的压缩包？

## 题目简述

`flag.zip` 的压缩数据实际上仍带有传统 ZipCrypto 的 12 字节加密头，但 ZIP 本地文件头和中央目录中的“通用位标志”被清零，解压软件因而把密文误当作普通 Deflate 数据。本题需要恢复两个标志字段，再使用 ZIP 注释中的密码解压。

## 解题过程

在文件末尾的 EOCD 注释中可以直接读到：

```text
I don't have a good memory, so I'll put the password here
password:0xGame2024
```

ZIP 对同一文件保存两份元数据：一份位于本地文件头 `PK\x03\x04`，另一份位于中央目录文件头 `PK\x01\x02`。原文件中两处 16 位“通用位标志”都是 `0x0000`，加密位没有置位。

分别把两处字段改成小端字节 `09 00`，即数值 `0x0009`：

- bit 0 置 1，表示文件使用传统 ZIP 加密；
- bit 3 置 1，使解密头按 DOS 修改时间校验，这与该密文原本的生成方式一致。

只改成 `0x0001` 会导致口令校验失败，因此两位都要恢复；本地文件头和中央目录也必须保持一致。

保存后用密码 `0xGame2024` 解压 `flag.txt`，得到：

```text
0xGame{M@ybe_y0u_ar2_t4e_mAsTer_0f_Z1p}
```

## 方法总结

这是 ZIP 元数据伪装，而不是密码爆破。检查此类异常 ZIP 时，应同时核对本地文件头和中央目录中的通用位标志、压缩大小以及文件尾注释；恢复正确的 `0x0009` 后，现成解压工具即可按 ZipCrypto 正常处理数据。
